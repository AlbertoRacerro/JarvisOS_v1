# 024 — FEM verification battery (analytic benchmark ladder)

Status: ready (drafted 2026-07-07 from expert kernel; expert review

where they conflict)

## Goal

After this slice, JarvisOS has an analytic verification ladder for the
mesh→solve chain: a small set of benchmark cases with known closed-form
solutions, each meshed by the 008 Gmsh adapter and solved by the 009 CalculiX
adapter, compared against the analytic value with a stated tolerance and pass
criterion. The battery runs as one command/test session under the existing
real-solver marker(s) and writes a persistent human-readable report under
`reports/`, following the pattern already established by
`reports/bluecad_spike_a1_a2.md` (generated by
`scripts/spikes/bluecad_spike_a1_a2.py`). This is what turns a `ResultSummary`
number into credible engineering instead of output from a black box, and it
re-runs whenever gmsh or CalculiX (or their adapters) are bumped.

## Why

`docs/strategy/PROGRAM_BACKLOG.md` row 024 and
`docs/strategy/JARVISOS_BETA_PROGRAM.md` Phase F row 024 both name this
battery as the professional-grade differentiator for BlueRev use — a solver
acceptance test, not a one-off spike. Spec 009 is static-only today (modal
and thermal are backlog item 027, "idea" status, not implemented); that
constraint is why battery v0 below is three static cases only, and why the
beam-frequency case both index rows still list is deferred rather than
included.

## Scope

In scope:

- Three static benchmark cases, each fully specified (geometry + material,
  BCs/loads, analytic formula with a literature reference, mesh prescription,
  target quantity, tolerance, pass criterion) per kernel decision 3 below.
- Benchmark geometry constructed directly at the mesh layer: each case
  supplies its own STEP file and a hand-authored/synthetically-generated
  manifest (`{"resolved_ports": {...}}` shape, the same shape
  `mesh_adapter.mesh_analysis_spec` already consumes via
  `analysis_spec["geometry"]["step_path"]` /
  `["manifest_path"]`) — no dependency on the BLUECAD L0/L1 CAD-generation
  path (`builders.py`, `assembly.py`, `loop.py`). This verifies the mesh→solve
  chain, not the CAD chain, per kernel decision 4.
- A comparison/report module with pure functions (analytic formulas, error
  computation, pass/fail evaluation) that has NO dependency on real gmsh/ccx
  binaries, so it can be unit-tested in the normal per-commit suite.
- A runner (pytest session marked with the real-solver marker, or a script
  under `scripts/`, implementer's choice — but it must be a single command)
  that, for each case: builds the STEP+manifest fixture, calls
  `mesh_adapter.mesh_analysis_spec`, then `fem_adapter.solve_static_analysis`,
  computes the analytic reference value, compares against the tolerance, and
  records a per-case verdict.
- A persistent human-readable report written under `reports/` (e.g.
  `reports/bluecad_fem_verification_battery.md`, matching the
  `bluecad_spike_a1_a2.md` convention: environment/tool versions, per-case
  table with computed vs. analytic value, relative error, tolerance, pass/fail,
  and artifact paths for the underlying mesh/solve run directories).
- Case (b) is marked "pending expert re-decision (linear elements)" per
  kernel decision 2(b) and the STOP finding below — it is fully specified
  except for its tolerance/mesh-order decision, which this draft does not
  make.

Out of scope (binding non-goals):

- No adapter feature additions to 008/009 (no new load/BC kinds, no new mesh
  controls, no element-order flag added to the mesh adapter's gmsh
  invocation).
- No modal, thermal, or buckling analysis (case (d), beam natural frequency,
  is deferred — see below).
- No UI.
- No per-case solver tuning beyond what `AnalysisSpec.mesh` /
  `AnalysisSpec.pass_criteria` already expose (no custom ccx cards, no manual
  mesh editing after gmsh generation).
- No property-based testing (that is backlog item 022; this battery is a
  fixed, hand-picked benchmark ladder, not a generator).
- No approximation hacks: no replacing a required load/BC with a different
  one and calling it equivalent unless the substitution is native to what the
  adapter already documents as its v0 load-type behavior, with the
  equivalence stated in the spec (see case (c) below — using `force_total` as
  a uniform axial tension is a direct application of the documented v0
  approximation in `fem_adapter.py`'s module docstring, not a new hack).

## Benchmark cases (kernel decision 2, binding)

### Case (a): slender cantilever, tip deflection under end load

- **Geometry**: a straight prismatic cantilever beam, length `L`, uniform
  cross-section (e.g. solid rectangular or circular section), constructed as
  a single STEP solid with two port frames in its manifest — `fixed_end`
  (root face) and `tip_end` (free-end face) — so the existing
  `BC_<label>`/`LOAD_<label>` physical-group mechanism in `mesh_adapter.py`
  applies unchanged. Slenderness `L / h >= 20` (`h` = section depth) so
  Euler-Bernoulli beam theory holds.
- **Material**: steel — state `E` and `nu` explicitly in the case fixture
  (e.g. `E = 200000` MPa, `nu = 0.3`, consistent with the mm/N/MPa unit
  system `AnalysisSpec.material` already documents in
  `schemas/bluecad_analysis_spec_v0_1.schema.json`).
- **BCs/loads**: `bcs = [{"port_label": "<part>.fixed_end", "kind": "fixed"}]`
  (schema only allows `kind: "fixed"` — matches); `loads = [{"port_label":
  "<part>.tip_end", "type": "force_total", "force": [0, F, 0]}]` (transverse
  tip force via the documented v0 `force_total` uniform-nodal-force
  approximation).
- **Analytic formula**: `delta = F * L^3 / (3 * E * I)` (Euler-Bernoulli
  cantilever tip deflection under a concentrated end load), with `I` the
  section's second moment of area about the bending axis. Reference: e.g.
  Gere & Goodno, *Mechanics of Materials*, cantilever tip-load case; or
  Roark's Formulas for Stress and Strain, Table for cantilever beams,
  concentrated end load case.
- **Mesh**: linear tetrahedra (`C3D4`, confirmed — see element-type finding
  below), target size fine enough to resolve bending across the section
  (state the chosen `target_size` in the case fixture relative to `h`).
- **Target quantity**: `max_displacement.value` from `ResultSummary`
  (magnitude of max nodal displacement — acceptable proxy for tip deflection
  if the tip node dominates; the case fixture must confirm the max-|u| node
  is at or near the tip).
- **Tolerance**: 2% on tip displacement (kernel decision, binding).
- **Pass criterion**: `abs(computed - analytic) / analytic <= 0.02`.
- **Capability check**: fixed BC (native, `kind: "fixed"`) + `force_total`
  load (native) — both supported by 009 today. Not blocked.

### Case (b): Lamé thick-walled cylinder under internal pressure — PENDING EXPERT RE-DECISION (linear elements)

- **Geometry**: a thick-walled hollow cylinder, inner radius `a`, outer
  radius `b`, axisymmetric, as a single STEP solid with an inner cylindrical
  face physical group for the pressure load and an end-face (or equivalent)
  physical group for a fixed/symmetry BC.
- **Material**: steel, `E`/`nu` stated explicitly (same as case (a)).
- **BCs/loads**: `bcs = [{"port_label": "<part>.end_face", "kind":
  "fixed"}]`; `loads = [{"port_label": "<part>.inner_face", "type":
  "pressure", "pressure": p}]` — native `pressure` load type (DLOAD on
  faces), no approximation needed.
- **Analytic formula**: Lamé's thick-cylinder solution — hoop stress at the
  inner radius under internal pressure only:
  `sigma_theta(a) = p * (b^2 + a^2) / (b^2 - a^2)`. Reference: Lamé (1852);
  standard form in Timoshenko & Goodier, *Theory of Elasticity*, or Boresi &
  Schmidt, *Advanced Mechanics of Materials*, thick-wall cylinder chapter.
- **Target quantity**: `max_von_mises.value` is NOT the same quantity as
  `sigma_theta(a)` — the case comparison must extract hoop stress at the
  inner-radius node/element specifically, or convert via the known stress
  state at that location (radial stress = `-p` at the bore, axial stress per
  end condition, von Mises computed from the analytic principal stresses)
  before comparing to whatever 009's `ResultSummary` reports. **This detail
  is not resolved in this draft** — see STOP below.
- **Tolerance/mesh — STOP, kernel conflict found**: the kernel decision as
  given assumes quadratic (second-order) tetrahedra with a 5% tolerance.
  Verified against the actual code: `mesh_adapter.py`'s gmsh invocation is
  `run_tool("gmsh", ["-3", str(geo_path), "-format", "inp", "-o",
  str(inp_path), "-save_all"], ...)` — no `-order 2`, no
  `Mesh.ElementOrder` setting anywhere in `_geo_text()`. Gmsh's CLI defaults
  to first-order (linear) elements when no order is requested, and this is
  independently confirmed by both the offline test fixture
  (`backend/tests/bluecad/fixtures/fake_gmsh.py`) and the real
  `test_mesh_adapter.py`/`test_fem_adapter.py` fixtures, which all encode
  `*ELEMENT, TYPE=C3D4` — CalculiX's 4-node linear tetrahedron, never
  `C3D10` (10-node quadratic). Linear tets are known to be materially
  stiffer/less accurate in bending- and curvature-dominated stress fields
  (exactly the Lamé hoop-stress case) than quadratic tets at a given mesh
  density; a 5% tolerance assuming quadratic elements is very likely too
  tight for linear elements at a practical mesh size, but tightening the
  target mesh density to compensate is a mesh-cost/accuracy trade-off
  judgment, not something this drafting pass should decide unilaterally.
  **Per the binding instruction, this case is marked "pending expert
  re-decision (linear elements)"** rather than assigning a tolerance. The
  geometry, material, BC/load wiring, and analytic formula above are fully
  specified and unaffected by this open point; only the tolerance value (and
  possibly a required mesh-refinement level near the bore to compensate for
  linear elements) is open.
- **Capability check**: fixed BC (native) + `pressure` load on a face
  (native, `DLOAD`) — both supported by 009 today. Not blocked by adapter
  capability; blocked only on the tolerance/element-order decision above.

### Case (c): plate with central hole in tension (Kirsch, finite-width correction)

- **Geometry**: a rectangular flat plate of width `W`, thickness `t`, with a
  central circular hole of diameter `d`, loaded in uniaxial tension along its
  length. Two port-frame physical groups: one loaded edge (`LOAD_<label>`,
  tension) and the opposite edge fixed (`BC_<label>`).
- **Material**: steel, `E`/`nu` stated explicitly.
- **BCs/loads**: `bcs = [{"port_label": "<part>.fixed_edge", "kind":
  "fixed"}]`; `loads = [{"port_label": "<part>.loaded_edge", "type":
  "force_total", "force": [F, 0, 0]}]` — uniform axial tension applied as a
  `force_total` distributed nodal load across the loaded edge's node set.
  This is a direct, documented use of 009's native v0 load type (uniform
  tension on an edge is exactly what `force_total`'s "divide total force
  across the target node set" semantics model), not a hack requiring a
  separate error-bound justification.
- **Analytic formula**: Kirsch's infinite-plate solution gives a theoretical
  stress concentration factor of 3.0 at the hole edge, but that value is only
  valid for `d/W -> 0`. This case uses the **finite-width-corrected** stress
  concentration factor `Kt(d/W)`, per Peterson's Stress Concentration
  Factors (Pilkey & Pilkey) chart/curve-fit for a central circular hole in a
  finite-width tension plate (commonly cited alongside the Heywood
  finite-width correction). The case fixture must cite the exact curve-fit
  formula or tabulated value used for the chosen `d/W` ratio, not the naive
  `Kt = 3.0`. Target quantity: `sigma_max = Kt(d/W) * sigma_nominal`, where
  `sigma_nominal = F / ((W - d) * t)` (net-section nominal stress, the
  convention Peterson's `Kt` charts use — the case fixture must state whether
  its chosen `Kt` source is net-section or gross-section based, since this
  changes `sigma_nominal`'s denominator).
- **Mesh**: linear tetrahedra (`C3D4`, confirmed — see finding below), with
  local refinement at the hole boundary (`AnalysisSpec.mesh.refinements`,
  already supported per-label) since stress concentration accuracy is
  mesh-density-sensitive at the hole edge.
- **Target quantity**: `max_von_mises.value` at (or near) the hole edge,
  compared against `sigma_max` above — for a uniaxial stress state at the
  hole boundary (radial stress = 0 at a free edge), von Mises reduces to
  `|sigma_theta|`, so this comparison is valid provided the max-von-Mises
  node is confirmed to be at the hole boundary, not elsewhere in the plate.
- **Tolerance**: 7% (kernel decision, binding).
- **Pass criterion**: `abs(computed - analytic) / analytic <= 0.07`.
- **Capability check**: fixed BC (native) + `force_total` load (native) —
  both supported by 009 today. Not blocked.

### Case (d) — DEFERRED: simply-supported beam, first natural frequency

Named here for completeness, not implemented in this slice. Trigger to
promote: **when spec 027 (modal analysis) lands** — 009 today only supports
`analysis_type: "static"` (`schemas/bluecad_analysis_spec_v0_1.schema.json`
`analysis_type.enum == ["static"]`, and `fem_adapter.solve_static_analysis`
hard-returns `SOLVE_ERROR` for anything else). A natural-frequency benchmark
requires an eigenvalue/modal solve, which does not exist in this codebase
yet. When 027 lands, this case should be drafted as a fourth verification
case (analytic reference: `f1 = (pi / 2) * sqrt(E * I / (rho * A * L^4))` for
a simply-supported uniform beam, or the standard Blevins/Timoshenko
closed-form for the first bending mode) with its own tolerance decided at
that time, not inherited from this draft.

## Element-type finding (binding on case (b) and case (c))

Verified against `backend/app/modules/bluecad/mesh_adapter.py`: the only
gmsh invocation is

```
run_tool("gmsh", ["-3", str(geo_path), "-format", "inp", "-o", str(inp_path), "-save_all"], attempt_dir, timeout_s, registry_path)
```

There is no `-order 2` argument, no `SetOrder(2)` / `Mesh.ElementOrder`
statement anywhere in `_geo_text()`. Gmsh's default element order is 1
(linear) when unset. This is corroborated by every `.inp` fixture in the
test suite: `backend/tests/bluecad/fixtures/fake_gmsh.py` and the golden
`.inp` text in `backend/tests/bluecad/test_fem_adapter.py`'s `_mesh()`
helper both hard-code `*ELEMENT, TYPE=C3D4` (CalculiX's linear 4-node
tetrahedron) — never `C3D10` (10-node quadratic tetrahedron). **The mesh
adapter produces linear tetrahedra only, not quadratic.** This is the basis
for marking case (b) "pending expert re-decision (linear elements)" per
kernel decision 2(b), and for stating the same fact as a caveat on case (c)'s
mesh prescription (finite-width Kirsch stress concentrations are also
gradient-sensitive, though case (c)'s 7% tolerance is the given kernel
decision and is not reopened here — only case (b)'s 5%-assuming-quadratic
tolerance is explicitly conditioned on element order in the kernel
instructions).

## Review resolutions (2026-07-07, expert review — supersede the body where they conflict)

1. **Element order: the whole battery runs on quadratic tetrahedra
   (`C3D10`), resolving the case (b) STOP.** Verifying stress against
   analytic solutions with linear tets means either loosening tolerances
   past credibility or paying for extreme mesh density — both defeat the
   battery's purpose. Therefore this spec's implementation includes, as its
   first slice, a narrowly-scoped **additive amendment to 008**: an optional
   `mesh.element_order` field in the AnalysisSpec schema, enum `[1, 2]`,
   default `1` (all existing behavior unchanged); when `2`, the gmsh
   invocation gains `-order 2`. Verify at implementation that
   `fem_adapter.py` passes `C3D10` elements through to CalculiX untouched
   (ccx supports C3D10 natively); if the adapter parses or filters element
   types anywhere, STOP and report. The binding non-goal "no adapter feature
   additions" is amended with exactly this exception and nothing else. A
   dated pointer to this resolution is added to spec 008.
2. **All three cases prescribe `element_order = 2`.** The body's "linear
   tetrahedra (`C3D4`, confirmed)" mesh lines for cases (a) and (c) are
   superseded. Tolerances confirmed: (a) 2%, (b) 5%, (c) 7%.
3. **Case (b) is un-blocked with tolerance 5%** under this mesh
   prescription: `b/a ≈ 2`, at least 3 elements through the wall thickness,
   cylinder length `L ≥ 3·b` so the fixed-end disturbance (Saint-Venant)
   decays before mid-length. Acceptance criterion 2 is replaced by: case (b)
   implemented and passing its 5% tolerance via the extraction in
   resolution 4; the required tests line "(a, c — b is blocked)" becomes
   "(a, b, c)".
4. **Case (b) extraction: battery-side solver-output sampling, not
   `max_von_mises`.** The draft's finding is correct — `max_von_mises` may
   sit at the fixed-end disturbance, not the bore. The comparison module
   parses the solver output (`.frd`) already produced in the run directory,
   samples the stress tensor at the node ring nearest `(r = a, z = L/2)`,
   and compares hoop stress (or von Mises computed from the sampled tensor
   against the analytic open-end state `σθ = p(b²+a²)/(b²−a²)`, `σr = −p`,
   `σz ≈ 0`). This is battery-owned parsing of an artifact the adapter
   already writes — no adapter change. If the run directory does not retain
   the `.frd`, STOP and report.
5. Cases (a) and (c) keep their `ResultSummary`-level extraction as drafted
   (max displacement / max von Mises), with their stated location-sanity
   checks.
