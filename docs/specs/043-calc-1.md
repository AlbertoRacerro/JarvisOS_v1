# 043 — CALC-1: runner extension for small engineering calculation scripts

Status: ready (after 016 is merged, and after 040)
Depends on: 016 (must be implemented and merged before this spec starts — it
provides the AST-allowlist mechanism this spec reuses; do not rebuild it
here), 040 (provides the memory facade this spec consumes)

## Goal

After this slice, the existing local Python runner can register and execute one
additional implementation kind, `calc_v0`, for small, bounded engineering
calculation scripts supplied as script text. Successful runs produce a
unit-carrying `result.json` persisted as a run artifact, and the runner service
calls the spec-040 memory facade to create `'proposed'`-status parameter
records with full provenance for each declared output. `batch_growth_v0`
behavior is preserved byte-for-byte.

## Why

JarvisOS needs a narrow, auditable path from AI-proposed engineering
calculations (buoyancy, mass balance, simple sizing, etc.) to reviewable
parameter records, without granting scripts database access or inventing a new
execution spine. The runner already owns script storage, SHA checks, run
directories, cleared subprocess environments, and artifact persistence (spec
016 established the pattern of adding a second, narrowly-scoped
`implementation_kind` with an AST import allowlist alongside the textual
denylist). This slice reuses that pattern for `calc_v0` and adds a mandatory
per-output `unit` field as the deliberate seed of a future units/sanity gate:
requiring a declared unit on every numeric output today is what makes a later
dimensional-consistency or unit-conversion check possible at all, without
retrofitting unitless historical data. Results become `'proposed'` parameter
records (never auto-accepted) so a human or deterministic policy must promote
them, per `AGENTS.md` invariant #8; the runner-to-memory link records
`origin='calc'` and the owning runner job id as `source_ref` so every
proposed value is traceable back to the exact script hash and inputs that
produced it, per invariant #9 (no fabricated results).

## Scope

In scope:
- Add `implementation_kind = "calc_v0"` alongside the existing
  `batch_growth_v0` (and, if spec 016 has landed by the time this is
  implemented, `bluecad_l2_v0`) path.
- For `calc_v0` registration only, accept caller-supplied Python script text,
  store it under the existing model implementation data-root location, compute
  SHA-256 with the existing artifact hash machinery (`sha256_file`), insert the
  script artifact, and link it from the model version, mirroring the
  `bluecad_l2_v0` registration flow in spec 016.
- Preserve `batch_growth_v0` registration semantics exactly: it still rejects
  arbitrary script text and still copies/uses the bundled batch-growth script.
- Define and validate the `calc_v0` input contract: `input.json` is a JSON
  object of named inputs, each shaped
  `{"value": <number>, "unit": <string>, "source_parameter_id": <string, optional>}`.
- Define and validate the `calc_v0` output contract: `result.json` declares
  named outputs, each shaped `{"value": <number>, "unit": <string>}` plus an
  optional `diagnostics` field; every output MUST declare a non-empty `unit`.
  An output missing `unit` (or with an empty/non-string `unit`) fails the job
  with a distinct, dedicated error code (not reused from any existing
  validation failure).
- Keep the existing textual denylist and add AST-based import allowlist checks
  for `calc_v0` scripts before execution, scoped to: `math`, `statistics`,
  `json` (stdlib only — see numpy finding below).
- Map every textual policy or AST allowlist failure for `calc_v0` to error
  type/code `SANDBOX_VIOLATION`, and make that failure non-retryable, matching
  spec 016's `bluecad_l2_v0` policy exactly.
- After a `calc_v0` run succeeds, store `result.json` as a run artifact through
  the existing declared-artifact registration path, then call the spec-040
  memory facade's in-process batch function once with all declared outputs, to
  create `'proposed'`-status parameter records with `origin='calc'` and
  `source_ref='runner_job:<runner job id>'` in a single transaction
  (all-or-nothing — see Design constraints).
- Add a determinism test: identical script hash + identical `input.json` byte
  content produce byte-identical `result.json` (a digest-comparison test, not
  merely equal parsed JSON).
- Add regression coverage proving `batch_growth_v0` externally observable
  behavior is byte-identical.

Out of scope (binding non-goals):
- No plotting, chart generation, or any binary/image artifact output.
- No file I/O beyond reading `input.json` and writing `result.json`; no script
  access to any other path, including run logs or other artifacts.
- No long-running or background jobs; `calc_v0` uses the existing synchronous,
  timeout-bounded subprocess execution path unchanged.
- No unit CONVERSION logic of any kind. This slice only checks that a unit
  string is present and non-empty on every output; it never parses, normalizes,
  converts, or validates dimensional consistency of unit strings. That is
  explicitly deferred to a future slice.
- No UI/frontend changes.
- No new dependencies. In particular, **no numpy** — verified absent from
  `backend/requirements.txt` (only `fastapi`, `uvicorn[standard]`, `pydantic`,
  `httpx`, `build123d`, `PyYAML` are listed). The `calc_v0` AST allowlist is
  therefore `math`, `statistics`, `json` only; numpy is a binding non-goal
  until a future spec adds it as a declared dependency.
- No implementation of spec-040's memory facade itself; this spec only
  consumes it as a contract.
- No changes to `batch_growth_v0` output schema, input validation rules, route
  behavior, status strings, log format, artifact rows, or bundled script.
- No OS-level sandboxing, containers, job objects, seccomp, or network
  namespaces beyond what the runner already does.
- No generic arbitrary Python runner; `calc_v0` is a fixed-contract kind for
  small declared-input/declared-output calculations only.

## Design constraints

- Honest isolation statement (verbatim, carried over from spec 016):

  > current runner isolation is not OS-level sandboxing; it is scoped scripts,
  > input validation, textual+AST checks, and a cleared environment. It must
  > not be described as network-secure. Stronger isolation
  > (job objects/containers) is a future, separate decision.
- `batch_growth_v0` is the regression anchor. Its behavior must remain
  byte-identical from the perspective of API responses, DB rows, copied script
  bytes, SHA-256 values, input validation failures, successful result
  payloads, logs, artifact declarations, and error/status strings.
- The `calc_v0` script source is an artifact, not executable mutable state.
  Job creation and execution must both continue to reject script bytes whose
  current SHA-256 differs from the stored artifact hash (same tamper-check
  pattern as `batch_growth_v0` and spec 016's `bluecad_l2_v0`).
- All new runtime paths continue to use data-root helpers from
  `backend/app/core/paths.py`; runtime data must not be written into the repo.
- The existing path constraints remain in force: implementation scripts stay
  under `model_implementation_root(workspace_id)`, run files stay under
  `run_root(workspace_id, simulation_run_id)`, and declared artifacts are
  relative paths under the run output directory.
- Subprocess execution remains `shell=False`, timeout-bounded,
  stdout/stderr bounded, and launched with the current cleared environment
  behavior (`execute_python_script`). Do not weaken environment clearing or add
  ambient secrets.
- **The sandboxed script never touches the database.** All parameter-record
  creation happens in the runner service, in-process, after the sandboxed
  subprocess has exited and `result.json` has been read and validated — never
  from inside the executed script.
- Policy checks are fail-closed: unparseable Python source, unknown import
  form, dynamic import, or any import outside the allowlist is a
  `SANDBOX_VIOLATION`.
- Keep the textual denylist (`FORBIDDEN_SCRIPT_MARKERS` in
  `backend/app/modules/runner/safety.py`) for `calc_v0` in addition to the new
  AST allowlist check, exactly as spec 016 requires for `bluecad_l2_v0`.
- AST import allowlist for `calc_v0`:
  - stdlib modules allowed: `math`, `statistics`, `json` only;
  - no third-party/project modules are allowed (unlike `bluecad_l2_v0`, which
    allows `build123d`);
  - numpy is explicitly NOT allowed (see numpy finding in Scope/non-goals);
  - do not allow `importlib`, `os`, `sys`, `subprocess`, `socket`, `urllib`,
    `http`, `httpx`, `requests`, filesystem mutation helpers, environment
    access, or any secret-related module/name.
- AST checks must inspect both `import x` and `from x import y`. Relative
  imports, star imports, `__import__`, `importlib`, `eval`, `exec`, and dynamic
  code loading are rejected — same rule as spec 016.
- Input contract for `calc_v0`:
  - `input.json` is a JSON object of named inputs; each named input is an
    object `{"value": <finite number>, "unit": <non-empty string>,
    "source_parameter_id": <string, optional>}`;
  - a missing or non-finite `value`, or a missing/empty `unit`, on any declared
    input fails job creation before a runner job is queued (reuse
    `canonical_json`/finite-number validation helpers from
    `backend/app/modules/runner/safety.py` for consistency with
    `validate_batch_growth_input`).
- Output contract for `calc_v0` successful runs:
  - `result.json` exists, is a JSON object, and declares named outputs, each
    `{"value": <finite number>, "unit": <non-empty string>}` plus optional
    `diagnostics`;
  - **every output's `unit` is mandatory.** A missing, empty, or non-string
    `unit` on any declared output fails the job with a distinct error code
    (e.g. `runner_output_unit_missing`), separate from
    `runner_result_invalid_json` and from any existing validation failure code,
    so this specific failure mode is machine-distinguishable from generic
    malformed-result failures;
  - `result.json` is additionally stored as a run artifact via the existing
    declared-artifact registration path (same mechanism `batch_growth_v0` uses
    for its `outputs/timeseries.csv` artifact declaration).
- Provenance and parameter-record creation:
  - after `result.json` passes output-contract validation and artifact
    registration succeeds, the runner service calls the spec-040 memory
    facade's batch function once with all declared outputs;
  - each created parameter record carries `origin='calc'` and
    `source_ref='runner_job:<runner job id>'` (spec 040's calc-origin
    contract), so the record is traceable to the exact script hash +
    input.json that produced it;
  - facade failure semantics are all-or-nothing (expert review, 2026-07-06):
    the batch call creates every output's record in one transaction; if it
    fails, the job is marked failed with a distinct error code (e.g.
    `runner_memory_facade_error`) and zero parameter records exist for the
    run — never a `succeeded` job with silently-dropped records.
- Determinism: for a fixed script SHA-256 and a byte-identical `input.json`,
  two separate runs must produce byte-identical `result.json` content
  (verified by comparing SHA-256 digests of the two `result.json` files, not
  just parsed-JSON equality, since the contract promises deterministic
  engineering calculations).
- `SANDBOX_VIOLATION` must be distinguishable from normal script runtime
  failure, validation failure, timeout, missing-unit failure, and artifact
  validation failure in runner responses/logs, and must be non-retryable —
  consistent with spec 016.
- Do not duplicate spec-040's parameter-record creation logic; call its
  exported facade function. If spec 040 does not yet expose an importable,
  stable facade at implementation time, stop and report the conflict rather
  than writing parallel parameter-insert SQL in the runner module.

## Acceptance criteria

1. `create_model_implementation(..., implementation_kind="calc_v0",
   script_text=...)` stores the provided script bytes under the workspace
   model implementation data-root path, records a `python_script` artifact
   with the correct SHA-256, and links that artifact from the created model
   version.
2. `batch_growth_v0` registration still copies the bundled script and
   ignores/rejects caller script text exactly as before; a golden regression
   test proves the copied bytes, SHA-256, returned fields, and relevant DB rows
   match the pre-change baseline.
3. Creating a `calc_v0` runner job with a valid input set (all named inputs
   carry finite `value` and non-empty `unit`) queues a job and writes
   `input.json` in the existing run-root layout.
4. Creating a `calc_v0` runner job with an invalid input set (missing/
   non-finite value, or missing/empty unit on any named input) fails before
   queueing a job and does not write run artifacts.
5. Running a safe `calc_v0` script (e.g. a buoyancy-force fixture: force =
   density × volume × g) that imports only `math`/`statistics`/`json`,
   reads `input.json`, and writes a fully unit-annotated `result.json`
   completes successfully, persists `result.json` as a run artifact, and
   creates one `'proposed'`-status parameter record per declared output, each
   with `origin='calc'` and `source_ref='runner_job:<job id>'`.
6. A `calc_v0` script whose `result.json` declares an output missing `unit`
   (or with an empty/non-string `unit`) is marked failed with a distinct error
   code that is not `runner_result_invalid_json` and not `SANDBOX_VIOLATION`,
   and no parameter records are created for that run.
7. A `calc_v0` script containing forbidden textual markers, disallowed imports
   (anything outside `math`, `statistics`, `json`), relative imports, star
   imports, `__import__`, `eval`, or `exec` is rejected before subprocess
   execution with `SANDBOX_VIOLATION`, and that failure is marked
   non-retryable.
8. A `calc_v0` script whose current bytes differ from the stored artifact SHA
   is rejected at both job creation and execution, matching the existing
   tamper-check pattern.
9. Running the same registered `calc_v0` script twice with byte-identical
   `input.json` produces two `result.json` files whose SHA-256 digests are
   equal.
10. The sandboxed `calc_v0` script process never opens a database connection
    or imports any DB-access module; all parameter-record creation happens in
    the runner service after subprocess exit, verified by the AST allowlist
    rejecting any DB/network-capable import and by test assertion that the
    subprocess environment excludes DB connection info.
11. Existing `batch_growth_v0` tests pass unchanged, and an added golden test
    demonstrates byte-identical behavior for successful run output and at
    least one representative validation/policy failure.
12. No test requires network access, live providers, a running Ollama
    instance, or OS-level sandbox features.

## Required tests

- Runner model registration tests:
  - `calc_v0` stores caller script text as a hashed artifact;
  - `batch_growth_v0` golden registration behavior remains byte-identical.
- Runner job creation tests:
  - valid `input.json` (finite values, non-empty units) queues successfully;
  - invalid input (missing/non-finite value, missing/empty unit) fails before
    queueing.
- Runner execution tests (happy path):
  - a fixture calc script (buoyancy force = density × volume × g, using only
    `math`) succeeds, produces a `result.json` artifact, and creates the
    expected `'proposed'` parameter records with `origin='calc'` and
    `source_ref='runner_job:<job id>'`.
- Runner execution tests (failure paths):
  - an output missing `unit` fails with a distinct, dedicated error code, and
    no parameter records are created;
  - a disallowed import (e.g. `os`, `subprocess`, or an un-allowlisted
    third-party module) fails with `SANDBOX_VIOLATION` before subprocess
    execution, and the failure is marked non-retryable.
- Determinism test: two runs of the same script hash with byte-identical
  `input.json` produce `result.json` files with equal SHA-256 digests.
- Tamper tests proving script SHA mismatch is rejected at job creation and at
  execution (mirrors the existing `batch_growth_v0`/spec-016 pattern).
- `batch_growth_v0` regression tests covering successful run output and a
  representative existing failure path; compare exact response/status/error
  payloads and relevant persisted values, excluding nondeterministic ids and
  timestamps only.

## Open questions

- **016 dependency status mismatch.** This spec was instructed to declare
  `Depends on: 016 (implemented), 040`, but as of drafting time
  `docs/specs/README.md` lists spec 016 status as `ready` (not
  `implemented`/`done`), and `backend/app/modules/runner/service.py` only
  wires `IMPLEMENTATION_KIND = "batch_growth_v0"` — there is no
  `bluecad_l2_v0` branch, no AST allowlist helper, and no `implementation_kind`
  dispatch logic anywhere in `service.py`/`safety.py`. In other words, the AST
  import-allowlist mechanism this spec depends on as a pattern to mirror does
  not yet exist in code; it exists only as the spec-016 design document. If
  016 has not landed by the time 043 is implemented, the implementer must
  build the AST-allowlist mechanism itself (generalized enough to support a
  second kind, or duplicated narrowly for `calc_v0`) rather than assume it is
  already present to extend. Maintainer should confirm whether "016
  (implemented)" is aspirational/planned-order or a documentation error.
- **Spec 040 does not exist yet.** `backend/app/modules/memory/` does not
  exist in the current tree, and no file matching `docs/specs/040*.md` exists.
  This spec was written against the *described* contract (a facade that
  creates `'proposed'`-status parameter records with `origin` and
  `source_ref`/`source_ai_job_id` provenance), per instruction to not read
  040's file since it is being drafted in parallel. Concretely open:
  - Exact facade function signature/module path (e.g.
    `backend/app/modules/memory/service.py:propose_parameter(...)` or
    similar) — unknown until 040 lands.
  - Exact provenance field name(s): the task context uses both `source_ref`
    and `source_ai_job_id` when describing 040 in general, but this task's own
    kernel decision #4 says "source_ref = runner job id" specifically. A
    runner job is not an `ai_jobs` row, so if 040's schema names the field
    `source_ai_job_id` and expects an AI job id specifically, `calc_v0`'s
    runner-job provenance may need a different field, a synthetic/bridging
    value, or 040 may need a runner-specific provenance variant. This must be
    reconciled against 040's actual schema once it lands.
  - Whether the current `parameters` table (schema migration
    `0004_engineering_record_schema_freeze`, spec 001) already has room for a
    `'proposed'` `value_status` and an `origin` column, or whether 040 is
    expected to add both. As verified in `backend/app/core/schema.py`, the
    current `value_status` enum (per spec 001) is `candidate`, `literature`,
    `measured`, `validated`, `accepted` — **`'proposed'` is not a currently
    valid value**, and there is no `origin` column on `parameters` today. This
    spec assumes 040 adds both; if 040 instead reuses `value_status='candidate'`
    with some other origin marker, this spec's acceptance criteria 5 and the
    "'proposed' parameter records" language throughout must be updated to
    match 040's actual shape.
  - Whether the facade call is transactional with artifact registration
    (all outputs succeed or the whole run fails) or best-effort per output.
    This spec requires "no silent drop" (see Design constraints) but the exact
    partial-failure semantics are 040's call to make, not this spec's.
- **numpy confirmed absent.** Verified against `backend/requirements.txt`
  (contents: `fastapi==0.115.6`, `uvicorn[standard]==0.34.0`,
  `pydantic==2.13.4`, `httpx==0.28.1`, `build123d==0.11.1`, `PyYAML==6.0.3`) —
  no numpy line exists. Per the binding kernel decision, numpy is therefore
  excluded from the `calc_v0` AST allowlist as a binding non-goal, not an
  oversight.
- **Interaction with a possible future third `implementation_kind`.** Spec 016
  already anticipates exactly one extension point
  (`batch_growth_v0` + `bluecad_l2_v0`); this spec adds a third
  (`calc_v0`). If 016 is implemented as a two-kind `if/elif` dispatch rather
  than a generalized kind registry, the 043 implementer may need to decide
  whether to generalize the dispatch (larger diff, more shared plumbing) or
  add a third parallel branch (smaller diff, more duplication). This spec
  intentionally does not mandate either approach — flag the choice made in the
  implementation notes.
